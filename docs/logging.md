# Logging System Architecture

EquiQuant implements a dual-layer logging infrastructure to separate development process multiplexing from application event tracing, and to safely bridge log output across parallel worker subprocesses.

# Log Destinations and Files

The application writes to five distinct log file categories based on how it is started and executed:

```text
+------------------------------------------------------------------------------+
| Log File Location   | Process / Command | Description                        |
+---------------------+-------------------+------------------------------------+
| logs/server.log     | API Backend       | Clean, structured JSON logs of the |
|                     |                   | Python backend application.        |
+---------------------+-------------------+------------------------------------+
| logs/orchestrator.log| make start        | Combined raw stdout/stderr of both |
|                     |                   | frontend (Vite) and backend.       |
+---------------------+-------------------+------------------------------------+
| logs/run_*.log      | CLI / scripts     | Clean JSON logs for a single run of|
|                     |                   | a script or CLI utility.           |
+---------------------+-------------------+------------------------------------+
| logs/ui.log         | make start/dev    | Segregated raw output from the     |
|                     |                   | React frontend Vite dev server.    |
+---------------------+-------------------+------------------------------------+
| logs/api.log        | make start/dev    | Segregated raw output from the     |
|                     |                   | backend FastAPI Uvicorn server.    |
+------------------------------------------------------------------------------+
```

## `logs/server.log`
This file is generated exclusively by the Python backend API. It uses a `RotatingFileHandler` to limit disk footprint (10MB limit with up to 20 backups). Logs are structured as single-line JSON records using `JSONFormatter`, which outputs properties such as timestamps, logging levels, logger names, message bodies, process IDs, and process names.

## `logs/orchestrator.log`
When you execute `make start` to launch EquiQuant, **Honcho** runs the Vite frontend and Uvicorn API server concurrently in the background. It redirects the combined `stdout` and `stderr` streams of both processes into `logs/orchestrator.log`. This log captures system start/stop lifecycles, Vite compiler status, and any raw `print()` or unhandled exception dumps that bypass the standard Python logging library.

## `logs/run_[timestamp].log`
Every script run (like database migrations or manual benchmark imports) initializes a unique, transactional log file. This prevents short-lived script runs from polluting the API server's persistent log file, making command-line run analysis straightforward.

## `logs/ui.log` and `logs/api.log`
These process-level log files are generated by **Honcho** when starting EquiQuant in either the foreground (`make dev`) or background (`make start`) using the `--logdir logs` option. 
- **`logs/ui.log`**: Contains only the frontend compiler updates, asset bundling progress, and eslint warning output.
- **`logs/api.log`**: Contains the raw Uvicorn server lifecycle updates, code reloading messages, and system standard output/stderr prints.

# Multiprocess Logging Pipeline

Parallel data fetching is run in worker processes spawned via `ProcessPoolExecutor`. Since child processes are spawned cleanly, they do not inherit active file handlers of the main process. 

To route worker logs back to the main logging pipeline without file lock contention or process blocks, EquiQuant implements a centralized, queue-based logging workflow:

* **Central Log Queue**: A thread-safe, multiprocessing Queue acts as the inter-process communication (IPC) channel.
* **Worker QueueHandler**: During subprocess startup, standard logging handlers are removed, and a `logging.handlers.QueueHandler` is attached. Any log messages emitted in the subprocess are serialized and sent over the queue.
* **Main Process QueueListener**: A background thread drains the queue in the main process.
* **Log Queue Dispatcher**: A custom `LogQueueDispatcher` forwards logs from the listener back into the main process's logging pipeline, letting them propagate normally to `logs/server.log` and console outputs.

## Logging Context
All logs format their origin process context:
* **`processName`**: Indicates whether it is the `MainProcess` or a child worker process (e.g. `SpawnProcess-1`).
* **`process`**: The operating system process ID (PID) of the source execution unit.
